⏳ ChronoSynth

Constraint-Aware Question Synthesis over Temporal Knowledge Graphs

ChronoSynth is a framework for temporal KG question generation with explicit temporal intent, indexed pattern memory, and constraint validation.

Feng Liang1, Weixin Zeng1, Runhao Zhao1, Mingjun Zhou1, Shaoshi Yang2, Xiang Zhao1,*

1National University of Defense Technology · 2Beijing University of Posts and Telecommunications

Overview

Temporal KG question generation is not only about fluency. A valid question must preserve the answer field, temporal rule, reference event, and ordering cue encoded in the input graph.

Temporal Intent State Represents answer field, temporal rule, reference point, and graph context before generation.
Indexed Pattern Memory Retrieves typed question patterns with temporal compatibility ahead of surface similarity.
Constraint Validation Filters answer leakage, unresolved placeholders, temporal-rule drift, and answer-field drift.

RQ2

Ablation: indexed memory is the main driver

Removing indexed pattern-memory retrieval causes the largest drop on both datasets. Relation-only retrieval recovers part of the gain, but the full rule-aware design remains clearly stronger on MultiTQ.

0.3042 -> 0.1449 MultiTQ BLEU-4 after removing full retrieval.
2.6172 -> 1.2250 MultiTQ CIDEr drop from full to no-memory.
0.3087 / 2.7035 100-sample MultiTQ result with full retrieval.
0.2635 / 2.2389 Relation-only retrieval still trails the full model.
RQ3

Robustness across temporal-semantic groups

The grouped analysis stays positive across temporal rules, graph sizes, and answer types. The improvements are therefore not limited to a narrow subset of easier temporal questions.

RQ4

Efficiency and indexed-memory reuse

ChronoSynth uses more tokens than the no-memory variant, but it preserves much better temporal quality. Once the index has been built, reuse sharply reduces the one-time overhead and improves practical efficiency across repeated runs.

1.790 s vs 4.302 s MultiTQ full vs no-memory latency in the runtime run.
699.20 vs 359.07 MultiTQ token budget for full vs no-memory.
74.183 s saved Construction time avoided by memory reuse.
35.06 MB Indexed memory size at full measured scale.
RQ5

Scalability with memory size

Scaling the indexed pattern memory increases construction time and storage, while quality remains comparatively stable once enough compatible patterns are available. The trend is visible in both BLEU-4 and CIDEr.

0.3380 BLEU-4 at 20% memory scale.
3.1234 CIDEr at 20% memory scale.
15.43 MB -> 35.06 MB Memory size from 20% to 100% scale.
15.126 s -> 74.572 s Construction time from 20% to 100% scale.
